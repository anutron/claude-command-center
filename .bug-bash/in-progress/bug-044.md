## Description
BUG-040 was supposed to change the default banner top padding from 10 to 2, but it still appears to default to 10. This is especially noticeable when settings get reset/blown away — the padding goes back to 10 instead of 2.

Double check that `GetBannerTopPadding()` actually returns 2 when the config field is nil/unset. There may be multiple code paths that set the default, or the change didn't stick.

## Expected Behavior
When `banner_top_padding` is not set in config.yaml, it should default to 2 lines of padding above the banner.

## Files Likely Involved
- `internal/config/config.go` — `GetBannerTopPadding()` method

## Resolution

BUG-040 correctly changed the default return value in `GetBannerTopPadding()` from 10 to 2, but missed updating the text input placeholder in the settings UI. The placeholder in `settings.go` line 160 was still `"10"`, which is shown as ghost text when the input field is empty. This made it appear that the default was still 10.

Fixed by changing the placeholder from `"10"` to `"2"` to match the actual default.

## Files Changed

- `internal/builtin/settings/settings.go` — Changed `pi.Placeholder` from `"10"` to `"2"`

## Choices Made

- Only changed the placeholder text since `GetBannerTopPadding()` already correctly returns 2 when the config field is nil/unset. The actual default logic was fixed in BUG-040; this was a cosmetic leftover.

## Uncertainties

- The placeholder is purely cosmetic (ghost text shown when the input is empty). The actual runtime default of 2 was already working correctly from BUG-040. If users still see padding of 10, it may be because their config.yaml has `banner_top_padding: 10` explicitly persisted from before BUG-040 was applied.
