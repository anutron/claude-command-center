## Description
Logs pane scrolling doesn't work at all — tested across two rounds. The filter (/) works fine but scrolling via j/k, arrows, ctrl+f/b/d/u does nothing. The key handlers exist but scrolling has no visible effect.

This has been reported twice now (BUG-028 and BUG-035 both attempted to fix it). The filter from BUG-035 works, but the scroll movement itself doesn't change what's displayed. The agent needs to actually trace why changing the scroll offset has no visible effect on the rendered output — likely the view function isn't using the offset to slice the log entries.

## Expected Behavior
- j/k and arrow keys scroll one line at a time
- ctrl+f/b/d/u scroll by page/half-page
- The visible logs should change when scrolling
- Scroll position indicator should update

## Files Likely Involved
Unknown — agent should explore `internal/builtin/settings/content_logs.go`. The key handlers update a scroll offset but the view function probably renders all logs without using the offset to window the output. Check if the view slices the log list by the scroll offset, and if the rendered height is being calculated correctly.

## Resolution

The view function and key handlers in `content_logs.go` were actually correct — the view properly windows log entries using `logOffset`, and the key handlers correctly update the offset. The test `TestLogsViewChangesWithScroll` confirms the view output changes after scrolling when in `FocusContent` mode.

**Root cause:** The bug was a focus-zone routing issue. When users navigate to the Logs item in the settings sidebar, they are in `FocusNav` mode. In that mode, j/k/up/down are consumed by the nav handler (`handleNavKey`) to move the sidebar cursor — they never reach the logs content scroll handler. The user would need to press Enter first to switch to `FocusContent` mode, but this wasn't obvious from the UI. The filter key (/) worked because unrecognized keys in `handleNavKey` are forwarded to the content handler via the `default` case — but j/k/arrows are recognized nav keys and never reach that forwarding path.

Previous fix attempts (BUG-028, BUG-035) correctly implemented the view windowing and key handlers for `FocusContent` mode, but missed that the keys were being intercepted by the nav handler in `FocusNav` mode.

**Fix:** Added an early intercept in `handleNavKey` that detects when the selected nav item is "system-logs" and forwards scroll keys (j/k/up/down/ctrl+f/b/d/u) directly to `handleLogsContentKey`, bypassing the nav cursor movement. This means the user can scroll logs immediately without needing to press Enter to switch focus zones.

## Files Changed

- `internal/builtin/settings/nav.go` — Added scroll key forwarding when logs pane is selected in FocusNav mode
- `internal/builtin/settings/settings_test.go` — Added `TestLogsScrollFromNavMode` to verify scrolling works from nav mode

## Choices Made

- Forwarded scroll keys to the content handler without switching `focusZone` to `FocusContent`. This keeps the nav sidebar visually focused while allowing scroll interaction with the content pane. The alternative (auto-switching to FocusContent) would prevent the user from navigating away from logs with j/k without pressing Esc first.
- Only applied the forwarding for the "system-logs" slug. Other content panes don't have scrollable content that conflicts with nav cursor movement.

## Uncertainties

- If more scrollable content panes are added in the future, the same forwarding pattern will need to be applied for those slugs.
- The `logsMaxVisible()` helper doesn't account for the filter line height, creating a minor mismatch with the view's `maxVisible` calculation when a filter is active. This could cause ctrl+f/b page scrolling to slightly overshoot, but the view's clamping corrects it. Not fixed here as it's a separate minor issue.
