## Description
Regression from BUG-008/BUG-017. In Calendar settings, pressing `f` does nothing visible — no loading spinner, no error message, no calendar list. The hint text "f fetch from Google" is visible but the key press has no observable effect. Screenshot shows the calendar settings page with "No calendars configured" and the hint line, but pressing `f` produces no feedback.

When a hotkey triggers an action that takes time, it should show a loading indicator. When it fails, it should show an error message. Currently neither happens — the user gets zero feedback.

## Expected Behavior
- Pressing `f` should immediately show a loading indicator (e.g., "Fetching calendars...")
- On success: show the calendar list with toggles
- On failure: show an error message explaining what went wrong (e.g., "Failed to fetch: invalid credentials")
- The UI should never silently do nothing on a keypress

## Files Likely Involved
Unknown — agent should explore `internal/refresh/sources/calendar/settings.go` for the `f` key handler and fetch flow. Check if the Cmd is being returned properly, if the loading state is being set, and if the view renders loading/error states.

## Resolution

The root cause of the `f` key not responding was fixed by BUG-022 (key routing from nav focus to content handler). After that fix, the fetch flow was already mostly wired up correctly:

- **Loading indicator**: `fetchLoading` is set to `true` immediately when `f` is pressed (line 352), and the view renders "Fetching calendars from Google..." (line 206).
- **Error display**: When the fetch fails, `fetchError` is set from `result.Err.Error()` and rendered as "Fetch error: ..." in red (line 209).
- **TeaCmd propagation**: The async fetch command is returned via `plugin.Action{TeaCmd: cmd}` and correctly propagated through `handleDatasourceContentKey` -> `handleContentKey` -> `HandleKey` -> TUI host, which executes it.

**One gap found and fixed**: After a successful fetch, the calendar settings did NOT auto-enter browse mode (`fetchMode`). The user would see the hint change to "f browse Google calendars" but would need to press `f` again. GitHub settings already had this auto-enter behavior. Fixed by setting `fetchMode = true` and `fetchCursor = 0` when calendars arrive successfully in `HandleSettingsMsg`.

## Files Changed

- `internal/refresh/sources/calendar/settings.go` — Auto-enter browse mode on successful calendar fetch (matching GitHub settings behavior)

## Choices Made

- Matched the GitHub settings pattern where successful fetch auto-enters browse mode, rather than requiring a second `f` press. This provides immediate feedback and is consistent across data sources.

## Uncertainties

- None. The fix is minimal and matches the existing GitHub pattern exactly.
