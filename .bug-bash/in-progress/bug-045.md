## Description
The Slack token setup flow now works (BUG-037 fixed the UI), but after saving a token and triggering a refresh, the Slack API returns `invalid_auth`. The token save UI works, but the token may not be read back correctly when making API calls, or there's a mismatch between where the token is saved vs where the Slack source reads it from.

Investigate the token flow: where is it saved in config.yaml, and where does the Slack refresh source read it from? Make sure they match. Also check if the token value is being trimmed of whitespace, and that the config is reloaded after save.

## Expected Behavior
After entering a valid Slack bot token via settings, running a refresh should successfully fetch Slack data (or return a meaningful error if the token itself is invalid, not a config wiring issue).

## Files Likely Involved
- `internal/config/config.go` — where token is stored
- `internal/refresh/sources/slack/slack.go` — where token is read for API calls
- `internal/builtin/settings/content_datasources.go` — where token is saved via UI

## Resolution

The token flow between settings save and Slack source read was correctly wired (both use `Slack.BotToken` / `slack.bot_token` in YAML), but the token was never trimmed of whitespace. When a user pastes a Slack bot token via the TUI form, trailing whitespace or newlines from the clipboard cause the `Bearer` header to contain invalid characters, resulting in `invalid_auth` from the Slack API.

Fixed by adding `strings.TrimSpace()` in two places:
1. `saveSlackToken()` in settings — trims before saving to config
2. `LoadSlackToken()` in config/validate — trims on read as defense-in-depth (covers both config file and env var paths)

## Files Changed
- `internal/builtin/settings/settings.go` — trim token before saving to config
- `internal/config/validate.go` — trim token when loading from config or env var

## Choices Made
- Applied trimming at both save and load points rather than just one, for defense-in-depth. The save-side trim ensures clean data on disk; the load-side trim handles env vars and any manual config edits.
- Did not add token format validation (e.g., checking `xoxb-` prefix) — the form description already hints at the expected format, and rejecting non-`xoxb-` tokens would prevent valid user tokens that use different prefixes.

## Uncertainties
- The pre-existing test `TestValidateSlack_MissingToken` was already failing before this fix (it doesn't isolate the config file from disk). This is unrelated to this change.
