## Description
BUG-046 fixed settings tests overwriting the real config by isolating them with CCC_CONFIG_DIR temp dirs. But the config is STILL being overwritten — user's banner name gets reset after running `make build && make install`.

`make build` runs `go build` and `codesign`. `make install` runs `ln -sf`. Neither should touch config. So either:
1. There's ANOTHER test file besides settings_test.go that calls config.Save() without CCC_CONFIG_DIR isolation
2. Something in the build process triggers config writes
3. A binary startup (ccc or ccc-refresh) writes defaults to config if certain fields are missing

Find EVERY call to `config.Save()` across the entire codebase (not just settings_test.go). Check each one:
- Is it in a test? If so, does it use CCC_CONFIG_DIR temp dir isolation?
- Is it in production code? Under what conditions does it trigger?

Also check if `make build` or `make install` runs any targets that execute Go tests or run the binary.

## Expected Behavior
No build, install, or test command should ever modify the user's config.yaml.

## Files Likely Involved
- `Makefile` — check all targets
- Search entire codebase for `config.Save(` calls
- Any test file that calls config.Save without CCC_CONFIG_DIR isolation

## Resolution

**Investigation findings:** Exhaustive search of all `config.Save()` call sites found 25+ calls in production code (settings, onboarding, nav toggles, data source settings) and confirmed that:

1. **Makefile is clean** — `make build` only runs `go build` + `codesign`, `make install` depends on `build` + `servers` (npm build), neither runs tests or the binary
2. **No test file directly calls `config.Save()`** besides `config_test.go` (already isolated) and `settings_test.go` (already isolated per BUG-046)
3. **No `init()` functions or `TestMain` functions** exist in the codebase
4. **Neither binary writes config on startup** — both `cmd/ccc/main.go` and `cmd/ccc-refresh/main.go` only call `config.Load()` which never writes to disk

**Fix:** Applied defense-in-depth `CCC_CONFIG_DIR` isolation to ALL test files that import or interact with the config package, even those that don't currently call `config.Save()`. This prevents any future code change from accidentally writing to the real user config through these test paths. The pattern ensures that if any tested function gains a `config.Save()` call path in the future, the test will safely write to a temp dir.

- `internal/refresh/sources/gmail/doctor_test.go` — added `CCC_CONFIG_DIR` to all 6 tests
- `internal/refresh/sources/gmail/gmail_test.go` — added `CCC_CONFIG_DIR` to all 4 tests
- `internal/refresh/sources/granola/doctor_test.go` — added `CCC_CONFIG_DIR` to both tests

## Choices Made
- Applied isolation broadly rather than only to tests that currently trigger `config.Save()`. This is a defense-in-depth strategy: if a function under test ever gains a Save call path, the test will be safe.
- Used `t.Setenv("CCC_CONFIG_DIR", t.TempDir())` consistently, matching the pattern established in BUG-046.
- Renamed `testConfig()` to `testSetup(t)` in `model_test.go` to match the convention used in `settings_test.go`.

## Uncertainties
- The exact reproduction path for the continued config overwrite is unclear. The exhaustive search found no unisolated `config.Save()` calls in any test file. The overwrite may have been caused by a transient state (e.g., a stale test binary from before BUG-046 was merged) or by running tests from a different working directory.
- Production code config.Save() calls in settings/onboarding are all user-initiated (key presses, toggle switches) and would only fire during interactive TUI use — not during builds or automated tests.
