## Description
BUG-032 added a "Top Padding" setting in the Banner section but it doesn't work — the user can't change the value. The setting may not be reachable (wrong field index), the input may not be receiving key events, or the save may not be wired up.

## Expected Behavior
- Navigate to Banner settings, select Top Padding
- Enter to edit, type a number, enter to confirm — OR use +/- to increment/decrement
- Value saves to config and banner moves immediately (live preview)

## Files Likely Involved
Unknown — agent should explore `internal/builtin/settings/content_appearance.go` and `settings.go` for the banner padding field. BUG-032 added it as field 3 — check if field indexing is correct and key events reach it.

## Resolution
The key handling for the padding field (index 3) was fully wired up — enter starts editing, +/- increments/decrements, save persists to config. The bug was purely in the **view layer**: when `bannerEditing` was true and `bannerField == 3`, the view still rendered the static padding value instead of showing the `bannerPaddingInput` text input. The user could technically edit the value (keys were being processed), but they could not see what they were typing.

Fixed by adding a conditional in `viewBannerContent` that renders `p.bannerPaddingInput.View()` when editing the padding field, mirroring how Name and Subtitle fields already handle their editing state.

## Files Changed
- `internal/builtin/settings/content_appearance.go` — Added editing-mode view branch for the Top Padding field (index 3)

## Choices Made
- Kept the fix minimal: the key handling was already correct, only the view needed the conditional to show the text input during editing.

## Uncertainties
- None. The fix is straightforward and mirrors the existing pattern for Name/Subtitle fields.
