## Description
The calendar `f` key now triggers a fetch (BUG-022 fixed the key routing), and the loading indicator shows "Fetching..." (BUG-019 verified), but the fetch never completes. The UI stays on "Fetching..." indefinitely.

The async fetch command likely fails silently — either the Google Calendar API call errors without sending a result message back, or the result message type isn't being routed to the calendar settings handler in Update.

Also: add logging to the plugin logger when the fetch starts, succeeds, or fails. Currently there's nothing in the logs to indicate anything happened.

## Expected Behavior
- Fetch should complete (success or error) within a few seconds
- On success: show calendar list
- On failure: show error message (e.g., "Failed: invalid credentials")
- Log entries should appear in the Logs tab for fetch start/success/failure

## Files Likely Involved
Unknown — agent should trace the async fetch Cmd in `internal/refresh/sources/calendar/settings.go`, check what message type it returns, and verify the settings plugin's Update routes that message type correctly. Also check if the Google Calendar API call itself is failing/panicking silently. Add logging via the plugin logger.

## Resolution
The root cause was in `internal/builtin/settings/settings.go` in the `HandleMessage` method. The provider message routing (which delivers `CalendarFetchResultMsg` to the calendar settings handler) was gated by `p.focusZone == FocusContent || p.focusZone == FocusEditing`. This meant:

1. If the user pressed `f` from the **nav sidebar** (which forwards keys to the content handler via the `default` case in `handleNavKey`), the fetch Cmd fired but `focusZone` remained `FocusNav`. When the `CalendarFetchResultMsg` arrived, it was silently dropped because the focus zone check failed.
2. If the user triggered a fetch from the content pane but navigated back to nav before the async result arrived, the result was also dropped.

In both cases, `fetchLoading` stayed `true` forever, showing "Fetching..." indefinitely.

The fix removes the focus zone restriction for provider message routing. Providers' `HandleSettingsMsg` only claims messages it owns (e.g., `CalendarFetchResultMsg`), so routing these regardless of focus zone is safe and correct.

Additionally, logging was added to the calendar settings fetch flow (start, success, failure) via the plugin logger.

## Files Changed
- `internal/builtin/settings/settings.go` — Removed focus zone gate from provider message routing in `HandleMessage`; pass logger to `calendar.NewSettings`
- `internal/refresh/sources/calendar/settings.go` — Added `logger` field and `log()` helper; added logging for fetch start (manual/auto), success, and failure; updated `NewSettings` to accept a `plugin.Logger`
- `cmd/ccc/main.go` — Updated `calendar.NewSettings` call to pass `nil` logger (doctor command doesn't need logging)
- `internal/builtin/settings/settings_test.go` — Added `TestCalendarFetchResultHandledFromNav` and `TestCalendarFetchResultErrorHandledFromNav` to verify the fix

## Choices Made
- **Removed focus zone restriction entirely** rather than adding `FocusNav` to the condition. The providers' `HandleSettingsMsg` is already selective about which messages it handles (type assertion), so the focus zone check was unnecessary for async results and only correct for key events (which use a separate code path).
- **Logger is optional (nil-safe)** — The `log()` helper checks for nil logger, so callers that don't need logging (like the doctor command) can pass nil.

## Uncertainties
- The fix routes provider messages regardless of focus zone, which means if the user switches to a different data source's settings while a calendar fetch is in flight, the message would still be routed to whatever provider is currently active (via `activeProvider()`). If the active provider is no longer calendar, the message would be passed to the wrong provider's `HandleSettingsMsg`, which would return `handled=false` and the message would be dropped. This is a minor edge case but could be addressed in the future by routing based on message type rather than active provider.
