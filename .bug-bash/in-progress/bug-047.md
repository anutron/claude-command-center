## Description
User enters a valid bot token (confirmed working via `curl -X POST https://slack.com/api/auth.test` with the same token). After saving it through the settings UI (press `a` on Slack data source), running a refresh still returns `slack API error: invalid_auth`.

BUG-045 added TrimSpace but that wasn't the real issue — the token itself is valid. The problem is somewhere in how the token flows from config.yaml to the actual Slack API call in the refresh source.

Possible issues to investigate:
- Is the token being saved to config.yaml correctly? (check the actual file on disk after save)
- Is `LoadSlackToken()` reading from the right config field?
- Is there an env var `SLACK_BOT_TOKEN` that's taking precedence with a stale/empty value?
- Is the refresh source using a different config instance than what was saved?
- Is the token being read before the config is reloaded after save?
- Does the Slack source actually use `LoadSlackToken()` or does it read the token from somewhere else entirely?

The screenshot shows: Status Disabled, Credentials incomplete, Last error: "fetch failed: slack API error: invalid_auth"

## Expected Behavior
After saving a valid bot token via settings UI and triggering refresh, Slack data should load successfully (or show a permissions error, not invalid_auth).

## Files Likely Involved
- `internal/config/validate.go` — LoadSlackToken()
- `internal/config/config.go` — Slack config struct, where token is stored
- `internal/refresh/sources/slack/slack.go` — where token is used for API calls
- `internal/builtin/settings/settings.go` — saveSlackToken() flow
- `cmd/ccc-refresh/main.go` — how config is loaded for refresh

## Resolution

The bug had two root causes:

### 1. Slack source ignored `cfg.Slack.Enabled` (primary flow issue)
Every other data source (calendar, GitHub, Gmail, Granola) receives the `enabled` flag from config and respects it in `Enabled()`. The Slack source was the only one that hardcoded `Enabled()` to `true` and didn't accept config at all. In `ccc-refresh/main.go`, the Slack source was created as `slacksrc.New(l)` without passing the enabled flag, unlike `granolasrc.New(cfg.Granola.Enabled, l)` etc.

### 2. `saveSlackToken()` didn't enable Slack in config
When the user saved a token via the settings UI, `saveSlackToken()` set `p.cfg.Slack.BotToken` but never set `p.cfg.Slack.Enabled = true`. Since `Enabled` defaults to `false` (Go zero value for bool), fixing issue #1 alone would have made the source never run. Both fixes are needed together: respect the flag AND auto-enable when saving a token.

### 3. Pre-existing test isolation bug
`TestValidateSlack_MissingToken` didn't set `CCC_CONFIG_DIR` to a temp directory, so `LoadSlackToken()` read from the real `~/.config/ccc/config.yaml` instead of an isolated test config. Fixed by adding `t.Setenv("CCC_CONFIG_DIR", t.TempDir())`.

### Token flow trace (end-to-end)
- Settings UI: `saveSlackToken()` -> sets `p.cfg.Slack.BotToken` + `p.cfg.Slack.Enabled = true` -> `config.Save(p.cfg)` -> writes to `~/.config/ccc/config.yaml`
- Refresh: `ccc-refresh` -> `config.Load()` -> `slacksrc.New(cfg.Slack.Enabled, l)` -> `Fetch()` -> `loadSlackToken()` -> `config.LoadSlackToken()` -> `config.Load()` (re-reads from disk) -> returns `cfg.Slack.BotToken`
- API call: token reaches `fetchSlackCandidates()` -> `Authorization: Bearer <token>` header

## Files Changed
- `internal/refresh/sources/slack/slack.go` — Added `enabled` field to `SlackSource`, `New()` accepts enabled flag, `Enabled()` returns it instead of hardcoded `true`
- `cmd/ccc-refresh/main.go` — Pass `cfg.Slack.Enabled` to `slacksrc.New()`
- `internal/builtin/settings/settings.go` — Set `p.cfg.Slack.Enabled = true` when saving a token
- `internal/config/validate_test.go` — Isolate `TestValidateSlack_MissingToken` with temp config dir

## Choices Made
- Auto-enable Slack when saving a token: when a user explicitly enters a token via the settings UI, the intent is clearly to use Slack. Setting `Enabled = true` automatically avoids requiring the user to also manually toggle the data source on.
- Aligned Slack source with other sources: every other source receives its `enabled` flag from config. The Slack source was the only outlier.

## Uncertainties
- The `search.messages` Slack API endpoint may require a user token (`xoxp-`) rather than a bot token (`xoxb-`). If the user is entering a bot token (as the UI suggests with "Slack Bot Token"), the API may return `invalid_auth` or `not_allowed_token_type` even after this fix. A follow-up may be needed to either change the API endpoint to one that works with bot tokens, or update the UI to request a user token instead.
